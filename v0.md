# 1) Confirm the container

- Verify ftyp includes an AVIF-compatible brand.
- Confirm meta exists and contains a primary item (pitm).
- Confirm the primary item’s type is av01.
- Confirm iloc points to a non-empty extent inside mdat.
- Confirm iinf/infe item IDs match the iloc item IDs.
- Confirm iprp/ipco/ipma exist if the file uses item properties. [paukerr.github](https://paukerr.github.io/av1-avif/)

If any of these fail, the decoder is not reaching AV1 at all, and the bug is in BMFF/HEIF parsing rather than image decode. [aomedia.googlesource](https://aomedia.googlesource.com/libavif/+/refs/tags/v1.1.0/include/avif/avif.h)

## 2) Check required AVIF properties

- av1C exists for the av01 item and is parsed correctly.
- ispe width/height match the final decoded image size.
- pixi channel count and bit depth match the actual pixel format.
- colr is handled if present, especially CICP/range signaling.
- Alpha items, if present, are parsed separately and tied to the color item correctly. [aomediacodec.github](https://aomediacodec.github.io/av1-avif/v1.2.0.html)

A lot of “wrong image” bugs are just property mismatches or missing property application. [fossies](https://fossies.org/linux/libavif/examples/avif_example_decode_file.c?m=b)

## 3) Validate the AV1 payload

- Dump every OBU type in the payload.
- Confirm the first OBU is a valid sequence header.
- Confirm OBU header fields are sane: no forbidden bit, valid size field, no truncated payload.
- Confirm the sequence header profile matches the AVIF profile.
- Confirm the sequence header’s coded dimensions are reasonable.
- Reject unsupported OBU types instead of silently misreading them. [github](https://github.com/cconcolato/av1-spec/blob/master/05.decoding.overview.md)

If the payload is malformed here, the decoder may produce corrupted output even if the container parser is correct. [storage.googleapis](https://storage.googleapis.com/oss-fuzz-coverage/libavif/reports-by-target/20251119/avif_fuzztest_dec@DecodeAvifTest.Decode/linux/src/libavif/src/reformat.c.html)

## 4) Compare against libavif behavior

Use libavif as a reference and compare:
- parse result,
- width/height,
- bit depth,
- yuv format,
- range,
- alpha presence,
- decode success/failure reason. [aomedia.googlesource](https://aomedia.googlesource.com/libavif/+/refs/tags/v1.1.0/include/avif/avif.h)

libavif exposes useful error categories such as BMFF parse failure, missing image item, decode color failure, decode alpha failure, color/alpha size mismatch, and ispe size mismatch. [github](https://github.com/AOMediaCodec/libavif/blob/main/src/avif.c)

## 5) Check YUV plane reconstruction

- Verify decoded Y plane size is exactly width * height.
- Verify U/V plane sizes match chroma subsampling.
- Verify alpha plane size matches the color plane size.
- Verify row stride is not accidentally treated as tight packing.
- Verify 4:2:0 chroma indexing uses the right subsampled coordinates.
- Verify monochrome files do not read invalid chroma pointers. [fossies](https://fossies.org/linux/libavif/examples/avif_example_decode_file.c?m=b)

If the image shape is wrong or strips appear, this is often a plane-size or stride bug, not a bitstream bug. [github](https://github.com/AOMediaCodec/libavif/blob/main/src/avif.c)

## 6) Check color conversion

- Use the right YUV matrix for the file’s metadata.
- Respect full-range vs limited-range.
- Expand 10-bit or 12-bit samples before converting to 8-bit RGBA.
- Handle alpha and premultiplication consistently.
- If the file uses HDR or wide gamut, avoid assuming default sRGB behavior. [storage.googleapis](https://storage.googleapis.com/oss-fuzz-coverage/libavif/reports-by-target/20251119/avif_fuzztest_dec@DecodeAvifTest.Decode/linux/src/libavif/src/reformat.c.html)

If the picture is recognizable but colors are wrong, the bug is almost always here. [bugzilla.mozilla](https://bugzilla.mozilla.org/show_bug.cgi?id=1731404)

## 7) Add targeted debug output

For each decode, print:
- file name,
- ftyp brands,
- pitm item id,
- iloc extents,
- av1C fields,
- ispe/pixi,
- OBU types and sizes,
- decoded plane dimensions,
- RGBA output dimensions. [aomediacodec.github](https://aomediacodec.github.io/av1-avif/v1.2.0.html)

That will let you compare your decoder line-by-line against libavif and isolate the first point where the two diverge. [fossies](https://fossies.org/linux/libavif/examples/avif_example_decode_file.c?m=b)

## 8) Fast triage by symptom

- Total garbage image: AV1 decode or OBU parsing bug.
- Correct image, wrong colors: range/matrix/chroma bug.
- Correct colors, wrong size: ispe/iloc/stride bug.
- Some files work, others fail: unsupported AVIF profile, depth, alpha, or color metadata. [aomedia.googlesource](https://aomedia.googlesource.com/libavif/+/refs/tags/v1.1.0/include/avif/avif.h)

## A practical debugging script

A very effective workflow is:
1. Decode with your library and dump metadata.
2. Decode with libavif and dump the same metadata.
3. Compare the dumps.
4. If metadata matches, compare Y/U/V plane hashes.
5. If planes match, compare RGBA bytes after conversion. [github](https://github.com/AOMediaCodec/libavif/blob/main/src/avif.c)
